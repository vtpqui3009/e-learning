## What is an API?

**API** (Application Programming Interface) is a way for applications to communicate with each other.

Think of it like a restaurant:
- **You** (the client) → Order food from a menu
- **Waiter** (the API) → Takes your request to the kitchen
- **Kitchen** (the server) → Prepares and returns your food

### REST API Basics

**REST** (Representational State Transfer) is the most common API style. It uses:

- **URLs** to identify resources
- **HTTP methods** to perform actions
- **JSON** for data format

## HTTP Methods (CRUD Operations)

| Method | Action | Example |
|--------|--------|---------|
| **GET** | Read data | Get all users |
| **POST** | Create new data | Create a new user |
| **PUT** | Update entire resource | Update all user fields |
| **PATCH** | Update partial resource | Update only email |
| **DELETE** | Remove data | Delete a user |

## HTTP Status Codes

Status codes tell you if your request succeeded or failed.

### Success Codes (2xx)

| Code | Meaning | When Used |
|------|---------|-----------|
| **200** | OK | Request succeeded |
| **201** | Created | New resource created (POST) |
| **204** | No Content | Success, no response body (DELETE) |

### Client Error Codes (4xx)

| Code | Meaning | When Used |
|------|---------|-----------|
| **400** | Bad Request | Invalid data sent |
| **401** | Unauthorized | Not logged in |
| **403** | Forbidden | No permission |
| **404** | Not Found | Resource doesn't exist |
| **422** | Unprocessable Entity | Validation failed |

### Server Error Codes (5xx)

| Code | Meaning | When Used |
|------|---------|-----------|
| **500** | Internal Server Error | Server crashed |
| **502** | Bad Gateway | Server communication failed |
| **503** | Service Unavailable | Server overloaded |

## Common Request Headers

| Header | Purpose | Example |
|--------|---------|---------|
| **Content-Type** | Format of data you're sending | `application/json` |
| **Accept** | Format you want back | `application/json` |
| **Authorization** | Authentication token | `Bearer eyJ...` |
| **User-Agent** | Client identification | `MyApp/1.0` |
| **X-API-Key** | API key authentication | `abc123xyz` |

## Manual Testing with Tools

### Tool 1: Using cURL (Command Line)

cURL is available on all operating systems:

```bash
# GET request
curl https://jsonplaceholder.typicode.com/users/1

# POST request with JSON body
curl -X POST https://jsonplaceholder.typicode.com/posts \
  -H "Content-Type: application/json" \
  -d '{"title": "Test", "body": "Hello", "userId": 1}'

# GET with headers
curl -H "Authorization: Bearer token123" \
  https://api.example.com/protected
```

### Tool 2: Postman (Recommended for Beginners)

Postman is a free GUI tool for API testing:

1. **Download**: https://www.postman.com/downloads/
2. **Install** and create free account
3. **Create new request**
4. **Enter URL and method**
5. **Add headers and body**
6. **Click Send**

### Tool 3: REST Client in VS Code

Install "REST Client" extension and create `.http` files:

```http

### Get single user
GET https://jsonplaceholder.typicode.com/users/1

POST https://jsonplaceholder.typicode.com/posts
Content-Type: application/json

{
  "title": "My Test Post",
  "body": "This is the content",
  "userId": 1
}
```

## Hands-On Practice: Task Manager API

Now let's apply what you've learned to the **Task Manager API** project!

### Setup

Make sure the Task Manager API is running:

```bash
cd sample-projects/TaskManagerAPI
dotnet run
```

Then open Swagger UI at `https://localhost:5001`

### Practice Exercise 1: Basic CRUD Operations

Test these operations using Swagger UI or Postman:

1. **GET all tasks**
   - Endpoint: `GET /api/tasks`
   - Expected: 200 OK with array of tasks
   - Verify: Response contains at least 3 seeded tasks

2. **GET single task**
   - Endpoint: `GET /api/tasks/1`
   - Expected: 200 OK with task details
   - Verify: Task has title, status, priority fields

3. **CREATE a task**
   - Endpoint: `POST /api/tasks`
   - Body:
     ```json
     {
       "title": "Learn Manual Testing",
       "description": "Practice with Task Manager API",
       "priority": "High",
       "userId": 1,
       "categoryId": 1
     }
     ```
   - Expected: 201 Created with task ID
   - Verify: Location header points to new task

4. **UPDATE the task**
   - Endpoint: `PUT /api/tasks/{id}` (use ID from step 3)
   - Body:
     ```json
     {
       "title": "Updated Title",
       "priority": "Critical"
     }
     ```
   - Expected: 200 OK with updated task

5. **DELETE the task**
   - Endpoint: `DELETE /api/tasks/{id}`
   - Expected: 204 No Content
   - Verify: GET same task returns 404

### Practice Exercise 2: Validation Testing

Test the validation rules:

1. **Empty title** (should fail)
   ```json
   {
     "title": "",
     "userId": 1
   }
   ```
   Expected: 400 Bad Request

2. **Past due date** (should fail)
   ```json
   {
     "title": "Test",
     "dueDate": "2020-01-01T00:00:00Z",
     "userId": 1
   }
   ```
   Expected: 400 Bad Request with error about past date

3. **Non-existent user** (should fail)
   ```json
   {
     "title": "Test",
     "userId": 999
   }
   ```
   Expected: 400 Bad Request with error about user not found

### Practice Exercise 3: Status Workflow Testing

Test the status transitions:

1. Create a task (starts as "Todo")
2. Update status to "InProgress":
   - Endpoint: `PATCH /api/tasks/{id}/status`
   - Body: `"InProgress"`
   - Expected: 200 OK

3. Update status to "Done":
   - Body: `"Done"`
   - Expected: 200 OK
   - Verify: `completedAt` field is now set

4. Try invalid transition (Todo → Done):
   - Create a new task
   - Try: `PATCH /api/tasks/{id}/status` with `"Done"`
   - Expected: 400 Bad Request (invalid transition)

### Practice Exercise 4: Filter Testing

Test the filtering capabilities:

1. **Filter by user**
   - `GET /api/tasks?userId=1`
   - Verify: All tasks belong to user 1

2. **Filter by status**
   - `GET /api/tasks?status=Todo`
   - Verify: All tasks have status "Todo"

3. **Get statistics**
   - `GET /api/tasks/statistics/1`
   - Verify: Counts match actual data

4. **Get overdue tasks**
   - `GET /api/tasks/overdue`
   - Verify: Only tasks with past due dates (none in seed data)

- **Always test**: Happy path, edge cases, errors, and validation
- **Document test cases** before writing automated tests
- **Practice with real projects** like Task Manager API
