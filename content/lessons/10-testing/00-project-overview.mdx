## Project Overview

## What Makes This Project Ideal for Testing?

### 1. Business Logic to Test

The project includes realistic business rules:

```csharp
// Validation: Due date cannot be in the past
if (request.DueDate.HasValue && request.DueDate.Value < DateTime.UtcNow.Date)
    throw new ArgumentException("Due date cannot be in the past");

// Validation: Title is required and has length limits
if (string.IsNullOrWhiteSpace(request.Title))
    throw new ArgumentException("Task title is required");

if (request.Title.Length > 200)
    throw new ArgumentException("Task title cannot exceed 200 characters");
```

### 2. Multiple Testing Levels

You can test at different levels:
- **Unit Tests**: Test individual methods in isolation
- **Integration Tests**: Test API endpoints end-to-end
- **Manual Tests**: Use Postman/Swagger to verify behavior

### 3. Real-World Scenarios

The project demonstrates common patterns:
- **CRUD operations** (Create, Read, Update, Delete)
- **Data validation** (required fields, data types)
- **Business rules** (status transitions, date validations)
- **Relationships** (tasks belong to users and categories)
- **Error handling** (appropriate HTTP status codes)

## API Endpoints

The API provides these endpoints for testing:

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/tasks` | Get all tasks (with filters) |
| GET | `/api/tasks/{id}` | Get specific task |
| POST | `/api/tasks` | Create new task |
| PUT | `/api/tasks/{id}` | Update task |
| DELETE | `/api/tasks/{id}` | Delete task |
| PATCH | `/api/tasks/{id}/status` | Update status |
| GET | `/api/tasks/statistics/{userId}` | Get statistics |
| GET | `/api/tasks/overdue` | Get overdue tasks |

## Business Rules to Test

The project implements several business rules that need testing:

1. **Task Creation**
   - Title is required
   - Title cannot exceed 200 characters
   - User must exist
   - Category must exist (if provided)
   - Due date cannot be in the past

2. **Task Updates**
   - Status transitions must be valid
   - Due date cannot be changed to past
   - Category must exist (if changing)

3. **Status Management**
   - Completing a task sets `CompletedAt`
   - Invalid transitions are rejected
   - Status workflow is enforced

4. **Statistics**
   - Counts are accurate
   - Overdue tasks are correctly identified
   - Per-user filtering works
